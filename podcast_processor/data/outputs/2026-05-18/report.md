### Prolog Coding Horror
*   Programas Prolog podem falhar em reportar soluções esperadas devido ao uso de construtos impuros e não-monotônicos como `!/0`, `(->)/2` e `var/1`. A alternativa declarativa é usar `dif/2`, `if_/3` e estruturas de dados limpas.
*   A modificação do banco de dados global com predicados como `assertz/1` e `retract/1` introduz dependências implícitas. A abordagem declarativa sugere passar o estado via argumentos de predicado ou notação de semicontexto.
*   A impressão de resultados diretamente no terminal (saída impura) impede o raciocínio sobre a saída, dificulta a escrita de testes e limita o uso do código como uma relação. Recomenda-se deixar o toplevel reportar as soluções ou usar `format_//2` para formatação pura.
*   O uso de construtos de baixo nível para aritmética, como `(is)/2`, `(=:=)/2` e `(>)/2`, torna a linguagem mais difícil de aprender e limita a capacidade de fazer consultas mais gerais. A alternativa declarativa é o uso de restrições CLP(FD), como `#=/2` e `#>/2`.

### An engineer’s guide to Model Context Protocol (MCP)
*   As interfaces do Model Context Protocol (MCP) devem ser de alta intenção e de chamada única, evitando estruturas de recursos aninhadas comuns em APIs tradicionais. As respostas devem incluir explicações estruturadas para falhas, permitindo a auto-correção eficiente dos agentes.
*   A autenticação para MCP deve seguir o modelo de delegação OAuth 2.0, utilizando tokens com escopo e autenticação baseada em sessão para granularidade, rastreabilidade e prevenção de acesso não autorizado por agentes.
*   A sanitização das respostas do MCP é crucial; elas devem conter apenas informações essenciais para o próximo passo do agente, evitando dados excessivos (como IPs ou logs verbosos) que podem gerar riscos de segurança ou aumentar a alucinação de LLMs.
*   Os servidores MCP devem manter os mesmos requisitos de Service Level Objective (SLO) para latência que as APIs empresariais, com chamadas tipicamente na faixa de dezenas a centenas de milissegundos, sendo menos gargalo que as chamadas a LLMs.
*   A avaliação da qualidade das interfaces MCP deve ser baseada em dados, como testes A/B, para determinar qual design produz resultados mais precisos e eficientes para os agentes.
